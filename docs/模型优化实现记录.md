### 作业目标

- 任务：Amazon Reviews 2023 上的 Top-K 商品推荐。
- 数据类别：`Industrial_and_Scientific`、`Musical_Instruments`、`CDs_and_Vinyl`。
- 主指标：测试集 `NDCG@10`。
- 提交内容：源代码、可执行程序结果文件、实验报告，报告中需注明最终 NDCG@10，且结果要可复现。

### 代码事实

- 项目基于 RecBole。
- 当前有两个自定义模型：`SASRecText` 和 `UniSRec`。
- 数据处理脚本用 `hyp1231/blair-roberta-base` 对商品文本元数据离线编码，保存为 `.feature` 文件。
- 模型训练阶段使用的是冻结的商品文本向量；用户侧没有独立文本/画像特征，只通过历史商品序列建模。
- `UniSRec` 当前配置为 `inductive_ft`，不会启用 trainable item ID embedding。

### 基线结果

| 模型 | 数据集 | 验证集 best NDCG@10 | 测试集 NDCG@10 | 测试集 Recall@10 | best epoch |
| --- | --- | ---: | ---: | ---: | ---: |
| SASRecText | Industrial_and_Scientific | 0.0296 | 0.0237 | 0.0456 | 93 |
| SASRecText | Musical_Instruments | 0.0349 | 0.0318 | 0.0600 | 85 |
| SASRecText | CDs_and_Vinyl | 0.0251 | 0.0238 | 0.0468 | 107 |
| UniSRec | Industrial_and_Scientific | 0.0322 | 0.0267 | 0.0524 | 167 |
| UniSRec | Musical_Instruments | 0.0374 | 0.0340 | 0.0650 | 128 |
| UniSRec | CDs_and_Vinyl | 0.0396 | 0.0368 | 0.0740 | 256 |

结论：`UniSRec` 是当前强基线，但训练耗时明显高于 `SASRecText`。

### 预算判断

- AutoDL 4090D 价格：1.88 元/小时。
- 20 元余额约等于 10.6 小时。
- 同学日志中 `UniSRec + CDs_and_Vinyl` 一次完整训练接近 16 小时，超过当前余额承受范围。

因此优化策略需要先小样本/短 epoch/小数据集验证，再扩大到三类数据。

### 镜像建议

优先选择 AutoDL 官方 `PyTorch 2.1.0 / Python 3.10 / CUDA 12.1` 镜像；如果不可用，选择 `PyTorch 2.1.2 / Python 3.10 / CUDA 11.8`。暂不建议选择 Python 3.12 系列镜像，以降低 RecBole 依赖冲突风险。

### 初始优化假设

1. `UniSRec` 当前强于 `SASRecText`，后续优先优化 UniSRec。
2. `transductive_ft` 可能通过启用 item ID embedding 提升三个固定测试类别上的效果，但会牺牲跨新商品归纳能力；本作业只评这三个固定类别，可以尝试。
3. `CDs_and_Vinyl` 训练很贵，优先在 `Musical_Instruments` 上快速筛选配置。
4. 完整联合训练语言模型不适合当前预算，先优化 adaptor、dropout、temperature、学习率和训练流程。

### 下一步计划

1. 在 AutoDL 上创建环境并验证 `python run.py -m UniSRec -d Musical_Instruments` 能启动。
2. 固定随机种子，先复现一条短训练结果。
3. 加入结果导出脚本或日志汇总脚本。
4. 开始第一组低成本实验：`train_stage=transductive_ft`、`temperature`、`dropout`、`learning_rate`。

## 2026-05-24 特征增强路线实现

### 动机

同学说明里提到当前模型主要是“物品侧特征用了预训练语言模型，用户侧还没用”。复查代码后确认：原始流程只把商品文本离线编码成 frozen PLM item embedding，用户侧主要是历史商品序列，未显式使用历史评分、时间间隔、商品价格、店铺、评分数、details 等信息。因此本轮不再只换序列模型，改为优先补强可用特征。

### 已完成代码改动

- `dataset/process_amazon.py`
  - 新增 `--output_domain`，便于生成不覆盖原数据集的 rich 版本。
  - 新增 `--rich_metadata`，rich 文本包含 `title`、`main_category`、`store`、`categories`、`features`、`description`、flattened `details`、price bucket、average rating bucket、rating-number bucket、best-seller-rank bucket。
  - 新增 `--with_user_features`，生成与 `item_id_list` 对齐的 `rating_list`、`time_gap_list`、`recency_list`；不使用目标商品的 rating/timestamp 作为输入。
  - 额外生成 `rich-item-struct.npy`，用于结构化 item metadata embedding。
- `data/dataset.py`
  - 新增 `RichUniSRecDataset`，加载 `rich-item-struct.npy` 并映射到 RecBole item id 空间。
- `model/richunisrec.py`
  - 新增 `RichUniSRec`。
  - item 表示由 rich PLM item embedding、trainable item ID embedding、结构化 metadata embedding 融合。
  - sequence 表示额外加入历史评分、时间间隔、recency embedding。
- `config/RichUniSRec.yaml`
  - 新增 rich 模型配置，`load_col` 包含 `item_id_list`、`rating_list`、`time_gap_list`、`recency_list`、`item_id`。
- `scripts/check_rich_dataset.py`
  - 新增 rich 数据校验脚本，检查三类 list 字段长度是否与 `item_id_list` 对齐。
